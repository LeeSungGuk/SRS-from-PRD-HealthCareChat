# SuperCalc 플랫폼 MVP 개발 목표 적절성 종합 검토 보고서
(난이도 / 가능성 / 효율성 중심)

## 1. 개요
* **검토 대상:** SRS v0.3 (건기식 성분·가격 비교 초자동화 플랫폼)
* **목적:** 초급 수준의 개발 지식을 가진 3년 차 IT 종사자가 '완전 바이브코딩' 방식으로 MVP를 개발할 때의 난이도, 시스템 효율성, 비용 적절성을 종합적으로 평가하여 안내.

## 2. 종합 평가 요약
현재 기획된 MVP는 **"완성형 서비스로서는 훌륭하지만, 바이브코딩으로 처음 개발을 시도하는 MVP로서는 범위가 너무 넓고 복잡합니다."**

* **개발 난이도 및 가능성 (★★☆☆☆):** 기획된 외부 API 연동(총 7개) 건수가 너무 많고, 실시간 계산 로직이 복잡하여 AI(바이브코딩)에 의존하는 개발 방식으로는 잦은 오류와 디버깅 늪에 빠질 확률이 매우 높습니다.
* **시스템 및 기술 스택 효율성 (★★★★☆):** 선택한 기술 스택(Next.js, Vercel, Supabase) 자체는 최신 트렌드와 바이브코딩에 아주 적합합니다. 한 곳에서 모든 것을 다 처리할 수 있어서 관리가 효율적입니다.
* **운영 비용 적절성 (★★★★★):** 클라우드 서비스의 프리 티어(무료 구간)를 최대한 활용하도록 설계되어, 초기 유지비용 측면에서는 최고 수준의 효율성을 보여줍니다.

## 3. 세부 영역별 검토

### 1) 개발 속도 관점 (MVP 단계의 과도한 시간 소요)
* **현재 문제점:**
  * 바이브코딩은 AI가 한 번에 코드를 잘 짜줄 때 매우 빠르지만, **여러 조건이 복잡하게 엉킨 로직**을 요구하면 AI가 혼동하기 시작하고, 이를 수정하는 데 기하급수적인 시간을 빼앗깁니다.
  * 특히 **3개 쇼핑몰(iHerb, 쿠팡, 네이버) 실시간 동시 조회 + 실시간 환율 적용 + 배송비/관세 처리 규칙**은 경우의 수가 너무 많아 AI에게 프롬프트로 완벽하게 텍스트로 설명하기가 대단히 어렵습니다.
  * **식약처 공공 API 실시간 연동** 및 데이터 파싱(분야에 맞는 분석) 부분도 예외 상황이 많아 예상치 못한 에러 처리에 대부분의 개발 시간을 소모할 수 있습니다.
* **권장 해결 방안 (Phase 0 제안):**
  * **초기엔 무조건 단순화:** 비교 쇼핑몰은 **가장 익숙한 쿠팡 1개**로 한정하여 먼저 로직을 완성하는 것을 목표로 합니다. 환율과 복잡한 관세 로직을 일단 덜어낼 수 있습니다.
  * **DB 수동/일괄 등록 (미리 채워넣기):** 초기 타겟 제품 300~500개는 API로 매번 검증해서 가져오지 말고, 엑셀 등으로 데이터를 정리한 후 DB(Supabase)에 한 번에 심어두는 방식이 속도 면에서 10배는 빠릅니다.

### 2) 외부 연동 목표와 기술 스택 (복잡성 및 리스크)
* **기술 스택 (Next.js, Prisma, Supabase, Vercel + shadcn/ui):**
  * **매우 훌륭합니다.** 이 조합은 AI(Claude, Cursor 등)가 가장 학습이 잘 되어 있는 최신 대세 스택입니다. 백엔드 서버를 따로 만들지 않고 하나의 코드 뭉치에서 모두 해결할 수 있어 바이브코딩에 최적화되어 있습니다.
* **외부 API 연동 리스크 (총 7종 연동 예정):**
  * **위험도(상):** 쿠팡/iHerb 쇼핑몰 API, 네이버 검색 API, 식약처 공공 데이터 API, 환율 API, 카카오톡 Link API, Gemini AI SDK 등 연동해야 할 외부 서비스가 지나치게 많습니다.
  * 각각의 API마다 '접근 권한(API Key) 환경설정', '하루/분당 호출 제한', '응답 지연에 따른 뻗음 현상(Time-out)' 등 신경 써야 할 설정이 다릅니다. 경험이 없는 상태에서 7개를 이어 붙이면 어디서 에러가 났는지 파악하기 어렵습니다.
* **권장 해결 방안:**
  * 연동을 최소화해야 합니다. 카카오톡 공유 기능은 카카오 SDK 환경설정(도메인 세팅 등)이 까다로우므로, 웹 브라우저의 기본기능인 **"URL 링크 복사"** 버튼으로 우선 대체하여 복잡도를 낮추는 것이 좋습니다.
  * 핵심 차별점이 될 AI 번역(Gemini) 연동 정도만 Vercel AI SDK를 이용해 집중적으로 구현해 봅니다.

### 3) 운영 소요 비용 (클라우드 컴퓨팅 서비스 기반)
* **현재 기획안:** 초기 서버 및 인프라 월 10만 원 이하 목표 달성 (Vercel, Supabase, Gemini 등의 무료 요금제 적극 활용).
* **운영 비용 장점 요약:**
  * 초기에 트래픽이 폭발하기 전까지는 Vercel(호스팅)과 Supabase(데이터베이스/스토리지)의 무료 구간 내에서 충분히 서비스 운영이 가능하여 **비용은 0원 내지 극소액**에 수렴할 것입니다.
* **숨은 비용 리스크 주의 (아키텍처 관점):**
  * Vercel의 Serverless Function(서버리스 함수)은 한 번 실행될 때 걸리는 '시간 제한'이 있습니다. 사용자가 검색할 때마다 3개 쇼핑몰 API를 실시간으로 동시에 긁어오면 지연시간(Timeout) 오류가 생길 수 있고, 잦은 호출은 요금제 초과로 이어집니다.
* **권장 해결 방안:**
  * 매번 남의 서버를 찌르는 '실시간 동시 연산'을 피하고, 배치(Batch) 작업처럼 주기적으로 가격 스냅샷을 우리 DB에 저장해두는 방식을 혼용하는 것이 유리합니다. 즉, 사용자에게는 **우리 DB에 이미 저장된 결과값을 빠르게 던져주는 구조**가 서버 비용 관리 및 반응 속도 측면에서 훨씬 좋습니다.

## 4. 결론 및 권장 액션 (바이브코딩 맞춤 전략)
현재 SRS의 기술적 설계 방향과 인프라 도구 선택은 흠잡을 데 없는 최상급 구조입니다. 다만, **'한 번에 구현하려는 기능의 양과 깊이(Scope)'**가 초급자의 바이브코딩 허용치를 넘어섭니다.

> 바이브코딩 개발로 지치지 않고 첫 MVP(가치 검증) 성공을 맛보려면, **가장 중요한 차별적 가치 한두 가지만 남기고 철저히 과감하게 쳐내야(Scope Down) 합니다.**

**🚀 바이브코딩 효율 극대화 MVP (슈퍼-라이트 버전) 제안:**
1. **타겟 쇼핑몰 구조조정:** '쿠팡' 단일 채널 연동으로 로직 대폭 간소화
2. **연동 최소화 및 수동화:** 식약처 뱃지 및 초기 성분 데이터 300건은 엑셀 데이터로 DB에 직접 삽입 (공공 API 실시간 연동 포기)
3. **공유/제보 간편화:** 카카오 API 대신 일반 '주소창 링크 복사하기' 사용. 오류 신고 센터는 시스템 내부에 구축하지 않고 '구글 폼(Google Forms)' 링크로 우선 연결.

위와 같이 목표를 조율한 후 "일단 굴러가는 프로토타입"을 일주일 안에 만들어보고, 그 위에 AI와 대화하며 하나씩 API를 추가 연동하는 점진적 방식을 강력히 추천합니다.
