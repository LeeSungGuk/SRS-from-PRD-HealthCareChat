## 4. Entity & Data Model 상세화

### 변경 내용

`6.2 Entity & Data Model`을 개념 엔터티 표 수준에서 구현 가능한 데이터 명세 수준으로 확장했다.

### 추가한 하위 섹션

- `6.2.1 Entity Summary`
- `6.2.2 Field-Level Schema`
- `6.2.3 Relationship & Cardinality Matrix`
- `6.2.4 Data Retention and Deletion Matrix`
- `6.2.5 Query Keys and Index Recommendations`
- `6.2.6 API Payload Schema Cross-Reference`
- `6.2.7 Enumerations`

### 보강한 내용

- Entity별 field type
- nullable 여부
- key / FK / index 기준
- relationship과 cardinality
- 데이터 유형별 보존/삭제 정책
- PoC report, usage metering, 운영 로그 조회를 위한 query/index 기준
- API endpoint와 backing entity 간 연결

### 핵심 판단

v0.4의 Entity & Data Model은 엔터티와 주요 필드 중심이었기 때문에 구현팀이 DB/API schema로 바로 전환하기에는 부족했다. v0.5에서는 SRS 수준에서 필요한 baseline schema를 제공하되, 물리 DB의 문자열 길이, partition strategy, storage engine은 상세 설계 산출물에서 확정하도록 남겼다.

### 추가한 Functional Requirements

### 주요 요구사항

- Partner Console 로그인 후 `tenant_id`, role, 접근 가능 모듈 표시
- API Key 발급/폐기/상태/마지막 사용 시각 관리

- Web API Playground에서 sandbox API 실행
- request builder validation
- sample elder profile과 정상/안전/오류 scenario 선택
- JSON response, HTTP status, latency, `request_id`, safety fields 표시
- cURL/Python snippet 생성
- PoC Report Console에서 기간 필터, 지표 조회, CSV/JSON export
- Ops Monitoring Console에서 로그, 오류, 안전 정책, 알림, 리뷰 큐 확인

- 보호자용/기관용/어르신용 완성 앱은 제공하지 않음

### 추가한 Non-Functional Requirements

- `REQ-NF-036`~`REQ-NF-041`

### 주요 NFR

- 모든 Partner Console 화면은 인증과 역할 기반 권한 검사 필요
- 대화 로그, 분석 결과, 리뷰 큐, 리포트는 기본 PII 마스킹
- 민감 UI 작업은 감사 로그 생성
- Playground 기본 실행 환경은 sandbox mode
- 주요 콘솔 화면은 baseline dataset 기준 p95 2초 이내 초기 조회 결과 표시
- CSV/JSON export는 원문/PII 제외 및 감사 로그 기록

### 핵심 판단

PRD v0.4의 Developer Portal, Sandbox, PoC Report, 운영 모니터링 요구는 화면 기반 제품 표면을 암시하고 있었다. 이를 보호자/기관용 완성 앱으로 확장하지 않고, B2B 개발자·PM·운영자를 위한 `SilverCare Partner Console`로 정리했다.

## 11. 단일 풀스택 MVP와 향후 분리 가능 제약 추가

### 변경 배경

MVP 단계에서는 구현 속도와 배포 단순성을 위해 Next.js 단일 풀스택 구조를 사용하되, 추후 프론트엔드와 백엔드를 분리할 수 있도록 문서상 제약을 명시해야 한다.

### SRS 변경

- `SRS/실버케어_SRS_v0.5.md`에 `1.5 Assumptions & Constraints` 추가
- `C-TEC-001`~`C-TEC-013` 추가
- 핵심 제약:
  - Phase A는 Next.js App Router 단일 풀스택 애플리케이션으로 구현
  - 별도 백엔드 서버는 Phase A 범위에서 제외
  - 서버 측 로직은 Server Actions 또는 Route Handlers로 구현
  - UI 및 스타일링은 Tailwind CSS와 shadcn/ui 사용
  - Client Component는 DB, LLM, secret, API Key 검증 로직을 직접 참조하지 않음
  - UI, API Route, domain service, repository/data access, LLM provider adapter, audit/logging 모듈 경계 유지
  - 외부 API와 Playground schema는 OpenAPI 호환 계약으로 관리
  - 로컬은 Prisma + SQLite 가능, 배포/PoC 검증은 Supabase PostgreSQL 기준
  - LLM 오케스트레이션은 별도 Python 서버 없이 Vercel AI SDK로 구현
  - LLM 호출은 Google Gemini API를 기본값으로 사용하고 환경 변수 기반 모델 교체를 지원
  - 배포와 인프라는 Vercel로 단일화하고 Git Push 기반 자동 배포를 사용
  - API 트래픽, background job, webhook retry, LLM orchestration, compliance boundary, 팀 소유권이 커지면 백엔드 분리 Phase 검토

### PRD 변경

- `실버케어초안_PRD_v0.5.md`의 ADR에 `ADR-007` 추가
- `9. SRS 전환 시 우선 확정해야 할 질문`에 단일 Next.js 구조의 분리 가능성 유지 질문 추가

### 설계 의도

현재는 통합 구조로 MVP를 빠르게 검증하되, API contract, service/repository boundary, LLM provider adapter, migration 기준을 문서상 제약으로 고정하여 추후 백엔드 분리 비용을 낮춘다.
